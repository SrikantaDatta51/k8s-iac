# Decoupling Strategy: Kubernetes from BCM
## The "Cut the Cord" Reference Architecture

### 1. The Strategic Goal
We are moving from a **Business Configuration Manager (BCM)** dependent model—where configuration is pushed imperatively from a central legacy tool—to a **Kubernetes Autonomous** model where the cluster pulls configuration from Git.

### 2. The "Strangler Fig" Pattern
We cannot shut down BCM overnight. We must "strangle" it by gradually moving responsibilities to Kubernetes.

| Layer | BCM Role (Initial) | Transition State | Final K8s Role |
| :--- | :--- | :--- | :--- |
| **Provisioning** | PXE, IPAM, RAID | BCM provisions bare metal; K8s takes over inside via CNI. | BCM only provides power/boot. K8s does the rest. |
| **app-config** | Pushes file to `/etc/app` | Script syncs BCM -> Git -> K8s ConfigMap. | `ConfigMap` managed purely in Git. |
| **Secrets** | Pushes `.pem` files | ExternalSecrets syncs from Vault. | ExternalSecrets + K8s Secrets. |
| **Networking** | Configures Host VLANs | Host uses flat bridge; Calico manages overlay. | Calico/Cilium fully autonomous. |
| **Monitoring** | Nagios Agent | Prometheus NodeExporter runs as DaemonSet. | Native Prometheus/Otel. |

### 3. The 4-Step Migration Process

#### Step 1: Inventory & Import (The Script)
*   **Goal**: Get the data out of BCM.
*   **Action**: Use the `generate_ssot_from_bcm.sh` tool.
*   **Outcome**: A set of `values.yaml` files in the Git repository representing the *current* productive state of BCM.

#### Step 2: The "Git Freeze"
*   **Goal**: Stop the drift.
*   **Action**:
    1.  Designate Git as the **Master**.
    2.  Configure BCM to **Read-Only** mode for application parameters.
    3.  If a change is needed, it goes to Git, then ArgoCD applies it.
    4.  *(Optional)*: Reverse-sync Git -> BCM if legacy reporting is needed.

#### Step 3: The Overlay Switch
*   **Goal**: Stop mounting host files.
*   **Action**:
    1.  Refactor Applications to read ConfigMaps/Secrets instead of `/etc/legacy-folder`.
    2.  Use `hostPath` volumes temporarily if code changes are expensive.
    3.  Deploy the "Uber Bundle" which abstracts the underlying infrastructure.

#### Step 4: The Disconnect (Day of Independence)
*   **Goal**: Verify Autonomy.
*   **Test**:
    1.  Block network access from BCM to the Kubernetes Cluster.
    2.  Trigger a Pod Restart.
    3.  **Success Condition**: The Pod comes up healthy without contacting BCM.
    4.  **Fail Condition**: The Pod hangs waiting for a license or config file from BCM.

### 4. Technical Decoupling Techniques

#### A. Networking (The "Bridge" Trick)
*   **Legacy**: BCM puts the server on a specific heavy VLAN with complex routing.
*   **Decoupled**: BCM sees a "Black Box". We use a Linux Bridge (`br0`) and run all logic inside the Software Defined Network (Calico).

#### B. Identity (The "OIDC" Shift)
*   **Legacy**: `/etc/passwd` synced by BCM.
*   **Decoupled**: K8s RBAC connected to OIDC (Okta/AzureAD). No local users generated by BCM are needed inside the cluster.

#### C. Compliance
*   **Legacy**: BCM scanner runs SSH loops to check versions.
*   **Decoupled**: Starboard/Trivy runs inside the cluster as a CRD, reporting compliance status to a central dashboard.
